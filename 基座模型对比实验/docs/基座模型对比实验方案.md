# 基座模型对比实验方案

## 背景与目标

已有5个Ollama本地模型（`qwen3:32b`, `deepseek-r1:70b`, `gpt-oss:20b`, `gemma3:27b`, `llama3.1:70b-instruct-q4_K_M`）在同一BIDS数据集（OHSU）上各自生成了C-PAC配置文件，输出格式为 `{dataset}_{model}_modifications.json` 和 `{dataset}_{model}_pipeline.yml`，存放在 `OUTPUT_ROOT_DIR/OHSU/` 下。

目标是写一个独立的评估脚本 `evaluate_models.py`，读取这些已有输出，计算以下5个指标，最终输出一张对比表格。

---

## 输入文件结构

```
OUTPUT_ROOT_DIR/OHSU/
├── OHSU_qwen3-32b_modifications.json
├── OHSU_qwen3-32b_pipeline.yml
├── OHSU_deepseek-r1-70b_modifications.json
├── OHSU_deepseek-r1-70b_pipeline.yml
├── ...（其余3个模型同理）
```

每个 `_modifications.json` 的结构大致为：
```json
{
  "config": {
    "modifications": [
      {"field": "...", "value": "...", "reason": "..."},
      ...
    ]
  },
  "metadata": {
    "model": "...",
    "timestamp": "...",
    ...
  }
}
```

---

## 需要计算的5个指标

**① 可执行率（Executable Rate）**
- 定义：pipeline.yml 是否能被 `yaml.safe_load()` 正常解析，且关键顶层字段存在（如 `pipeline_setup`）
- 计算：直接 try/except 加载yml，通过=1，失败=0

**② 字段合法率（Field Validity Rate）**
- 定义：modifications.json 中每条修改的 `field` 路径，能否在模板yml（`pipeline_config_default.yml`）中找到对应的键
- 计算：遍历所有modification条目，用点分路径在模板yml中查找，统计命中比例

**③ 回退率（Fallback Rate）**
- 定义：modifications中有多少条因字段不合法或值类型错误而未能成功写入最终yml（即"建议了但没生效"）
- 计算：对比 modifications.json 中的建议字段列表，与实际 pipeline.yml 相比模板yml发生变化的字段列表，统计未生效比例

**④ 平均修改字段数（Avg Modified Fields）**
- 定义：modifications列表的长度，即模型建议修改了多少个参数
- 计算：直接 `len(modifications)`

**⑤ 运行成本估算（Cost Proxy）**
- 定义：用 `metadata` 中记录的总token数（prompt_tokens + completion_tokens）作为成本代理指标；若json中没有token信息，则用修改建议的字符总长度作为替代
- 计算：优先读 metadata 中的 token 字段，没有则统计所有 modification 条目序列化后的字符数

## 脚本整体结构

```
evaluate_models.py
│
├── 读取配置：OUTPUT_ROOT_DIR、YML_TEMPLATE_PATH（从main.py复用路径常量）
│
├── 发现所有模型的输出文件对（json + yml）
│
├── 对每个模型依次计算5个指标
│   ├── load_and_validate_yml()       → 可执行率
│   ├── check_field_validity()        → 字段合法率
│   ├── compute_fallback_rate()       → 回退率
│   ├── count_modifications()         → 平均修改字段数
│   └── estimate_cost()              → 运行成本
│
├── 汇总为 DataFrame，打印表格
│
└── 保存为 model_comparison.csv 和 model_comparison.md（便于直接粘贴进论文）
```

---

## 注意事项给执行AI

1. **路径复用**：直接从 `main.py` 的常量里读 `OUTPUT_ROOT_DIR` 和 `YML_TEMPLATE_PATH`，不要硬编码
2. **字段路径解析**：C-PAC yml是深度嵌套结构，字段合法率的查找需要支持多级嵌套键（如 `functional_preproc.motion_estimates_and_correction.motion_correction.using`），建议写一个递归查找辅助函数
3. **回退率计算**：需要先对模板yml和生成yml做深度diff，找出实际发生变化的字段集合，再与modifications建议集合做对比
4. **输出格式**：最终表格每行一个模型，每列一个指标，数值保留两位小数，运行成本列注明单位（tokens或chars）
