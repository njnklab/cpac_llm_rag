# 基座模型评估指标计算细节与多次重复实验方案

## 一、 五大评估指标计算说明

在我们的 Python 脚本 (`evaluate_models.py`) 中，针对每个模型生成的 `_modifications.json`（大模型的修改建议）以及 `_pipeline.yml`（由更新程序最终合并出来的 C-PAC 配置文件），我们提取了五个量化指标，它们的具体计算逻辑如下：

### 1. 可执行率 (Executable Rate)
- **含义**：大模型生成的配置能否在后续流程（即 C-PAC）中被 Python 标准的 YAML 库正常解析，是否破坏了最核心的文件结构。
- **计算逻辑**：
  1. 尝试使用 `yaml.safe_load()` 读取生成的 `_pipeline.yml`。
  2. 检查解析出来的字典内是否包含最顶层的基础键，如 `pipeline_setup`。
  3. 如果读取不报错并且包含核心键，得分为 `1` (100%)，否则为 `0`。

### 2. 字段合法率 (Field Validity Rate)
- **含义**：考量大模型是否存在“幻觉”。大模型在 JSON 文件中建议修改的路径（`parameter_path`，如 `functional_preproc.motion_correction.using`），是否真正在原版 C-PAC 的模板 YAML (`pipeline_config_default.yml`) 中存在。
- **计算逻辑**：
  1. 遍历 JSON 中提供的每一次修改建议。
  2. 根据它提供的由“点”分隔的嵌套路径，在**原始的模板 YAML** 中进行逐层寻找。
  3. `合法率 = 可以在模板中找得到的路径数量 / 模型总共提出的修改建议数量`。

### 3. 回退率 (Fallback Rate)
- **含义**：模型所提出的有效修改建议中，有多少**未能**真正在最终生成的 YAML 中展现出来（例如建议了修改，但是因为值类型错误或其他原因导致被覆写，或者 C-PAC 更新函数拒绝了它）。
- **计算逻辑**：
  1. 对于模型提出的修改建议，比较该字段在“原模板 YAML”中的值，以及在“最终生成 YAML”中的值。
  2. 如果最终生成的值仍然等于原来的值（或者最终生成的 YAML 里根本没有这个键），说明没改动成功，这被视作一次回退 (Fallback)。
  3. `回退率 = 修改未生效的数量 / 模型总共提出的修改建议数量`。
  - *(注：由于 `update_yml_from_json.py` 工具当遇到不存在的键时默认会强行创建字典并写入，这导致非法键也会被写入最终YAML。因此在当前框架下，回退率大部分情况下是 0)。*

### 4. 平均修改字段数 (Avg Modified Fields)
- **含义**：衡量模型修改配置的**激进程度**。
- **计算逻辑**：直接统计模型输出的 JSON 中包含的 `modifications` 修改列表的长度（`len(modifications)`）。

### 5. 运行成本估算 (Cost Proxy)
- **含义**：衡量这个模型生成配置的“啰嗦程度”或者开销大小。
- **计算逻辑**：优先从 Ollama 返回的 metadata 中统计所有的 Token 消耗（把 Prompt Count 和 Completion Eval Count 相加）。如果没有 Token 数据，则转为统计整个修改块序列化成字符串后的字符总数（chars）。

## 二、 为什么要进行多次调用？ (实验设计的升级)

你提到：“实验是不是得多次调用模型？例如每个模型都调用 10 次 20 次？”
**这个思路非常正确且这是评测 LLM 时极具科研价值的一环！**

大模型（尤其是温度 Temperature > 0 时）具有**随机性**。在单次运行中，某个模型碰巧表现很好或者很差，不能说明它的真实水平。因此，针对你想要测试的新数据集 (`/mnt/sda1/zhangyan/openneuro/ds002748`)，你需要设计 **多次重复实验方案 (N Repetitions)**。

### 多次调用的核心评估维度：稳定性 (Robustness / Variance)
如果一个模型连续 10 次为同一个数据集生成的参数配置是高度相似的，说明它对特定任务的理解很稳健；如果 10 次给出了 10 种完全不同大相径庭的参数组合，说明它的不确定性极强，不可用于严谨的医学处理流程。

### 升级后的多轮实验方案流程设计：

1. **确定实验参数**
   - **数据集**: `/mnt/sda1/zhangyan/openneuro/ds002748`
   - **重复次数 N**: 建议设置为 `10` 次（10 次足以计算方差，且时间成本可以接受）。

2. **中间结果在对比实验路径中的保存规范**
   为了防止文件覆盖，原本的文件名加上轮次（`run_n`）后缀，全部汇总放入基座对比专门用于存放中间结果的文件夹中：
   ```text
   /home/a001/zhangyan/cpac/llm_parameters/20250825全参数整理/基座模型对比实验/ds002748_runs/
   ├── run_1/
   │   ├── ds002748_qwen3-32b_modifications.json
   │   ├── ds002748_qwen3-32b_pipeline.yml
   │   ├── ... (其他 4 个模型)
   ├── run_2/
   │   ├── ...
   ...
   ├── run_10/
   ```

3. **评估脚本的升级逻辑 (针对 N 轮求平均)**
   当这 10 轮调用的数据（即 $5 \times 10 = 50$ 组 JSON+YML 文件对）生成完毕后，我们的 `evaluate_models.py` 会做两件事：
   - 将上述的 5 个指标从“单点值”升级为**10次的平均值 (Mean) 和 误差棒 (Standard Deviation)**（例如：`合法率 0.85 ± 0.04`）。
   - **新增特殊指标：参数一致性 (Parameter Consistency Rate)**。即在这 10 次里，该模型要求修改的“平均并集”是什么？它对某一个参数的建议是否在 10 次里都是相同的？

你可以通过一个 Python 循环脚本来包装原本的 `generate_dataset_config.py`，让它自动完成这 10 次调用和分别建立带有 `run_x` 带有后缀的文件保存！
